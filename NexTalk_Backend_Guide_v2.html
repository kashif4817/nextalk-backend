Security & Performance — When to Add What

This is not a "later" topic. Some things must be done from Day 1 or your app will be broken by design.

Day 1 — Non-Negotiable

sendResponseConsistent JSON responses across all endpointsWrite once, use everywhere
Auth middlewareEvery protected route needs itVerify Supabase token → attach req.user

CORS configRestrict which origins can call your APIWhitelist your frontend URL only

Phase 4Cursor-based paginationNever return 10,000 messages at once

Phase 9 — Polish (After All Features Work)

  • Supabase Row Level Security (RLS) — database-level access control
  • Database indexes — speed up slow queries

  • Morgan logging — log all HTTP requests in production
  • CORS audit — remove any wildcard *, finalize allowed origins
The Rule Security at system boundaries (auth, input) = Day 1. Performance optimization = when you see slowness. Security audit = Phase 9. Never add caching until you actually need it.

Phase 1 · Week 1 · Days 1–4

Foundation & Authentication

Goal: Running Express server + Supabase connected + Auth working + Utilities ready

This is the skeleton of your app. A weak foundation means rewriting later. Take your time here.

Supabase Auth — How It Works

  1. User calls POST /api/auth/signup → backend calls supabase.auth.signUp()
  2. Supabase creates user in auth.users, sends confirmation email automatically
  3. User confirms email → account is active
  4. User calls POST /api/auth/login → backend calls supabase.auth.signInWithPassword()
  5. Supabase returns { access_token, refresh_token, user }
  6. Frontend sends token on every request: Authorization: Bearer <token>
  7. Your authMiddleware calls supabase.auth.getUser(token) to verify
  8. If valid → attach user to req.user → continue to controller
Supabase vs Custom JWT With custom JWT you call jwt.verify(token, secret). With Supabase you call supabase.auth.getUser(token). Supabase handles signing, expiry, and refresh internally. You do NOT manage JWTs manually.

Middleware Order in app.js (Critical)

1. helmet()           ← Security headers FIRST
2. cors()             ← Allow your frontend domain
3. morgan('dev')      ← Log all requests
4. express.json()     ← Parse request body
5. authRateLimiter    ← Only on /api/auth/* routes
6. routes             ← Your actual API routes
7. 404 handler        ← After all routes
8. global error handler ← LAST, catches everything
  • 🔴 CriticalInitialize project — npm init, install Express, nodemon, dotenv, helmet, cors, morgan
  • 🔴 CriticalCreate config/supabase.js — Supabase client as singleton, export once
  • 🔴 CriticalWrite asyncHandler.js — wraps async functions, catches errors, calls next(err)
  • 🔴 CriticalWrite sendResponse.js — consistent JSON: { success, message, data, statusCode }
  • 🔴 CriticalWrite authMiddleware.js — extract Bearer token → supabase.auth.getUser(token) → attach req.user
  • 🔴 CriticalBuild authController.js — register, login, logout, refreshToken, forgotPassword, resetPassword
  • 🟡 ImportantAdd rateLimiter.js — 5 requests per 15 min for all auth routes
  • 🟡 ImportantSet up global error handler in app.js (4-param function: err, req, res, next)
  • 🟢 NiceTest all auth routes in Postman
  • 🟢 Nicegit commit: feat: add foundation — server, supabase, auth, utilities then merge to main
Challenge — Phase 1 After building authMiddleware, think about this: what if the token is valid but the user was deleted from Supabase after getting the token? Does your middleware handle it? What should you return? Handle this edge case yourself before moving on.
Do NOT do this Never call createClient() inside a controller or middleware. Create the Supabase client ONCE in config/supabase.js and import it everywhere.

Phase 3 · Week 2 · Days 1–3

Conversations

Goal: Create 1-to-1 conversations, list all, get single conversation

The conversation_members junction table design is the key concept. Understand it before writing a single line.

The Junction Table Design

conversations tableconversation_members table
1 row: { id: "conv_123", is_group: false }2 rows:
{ conv_id: "conv_123", user_id: "userA" }
{ conv_id: "conv_123", user_id: "userB" }

Creating: insert conversation → insert 2 rows in conversation_members (you + receiver). Listing: find all rows in conversation_members where user_id = current user → get their conversation_ids → fetch those conversations.

Edge Case — Duplicate Conversations What if User A tries to start a conversation with User B when one already exists? Check first. If yes, return the existing one. Do NOT create a duplicate. Think about how to query this.

getAllConversations Should Return

For each conversation: the other user's name + avatar, last message preview (~60 chars), last message timestamp, unread count. Store last_message_preview and unread_count directly in the conversations table — update them every time a message is sent.

  • 🔴 CriticalcreateConversation — check for existing conversation between these two users first, then create
  • 🔴 CriticalgetAllConversations — return with last_message_preview + unread_count, sorted by last_message_at DESC
  • 🟡 ImportantgetSingleConversation — return conversation + all members, only accessible by members
  • 🟢 NiceTest in Postman then git commit: feat: add conversation endpoints
Challenge — Phase 3 In getAllConversations, conversations are sorted by most recent message (like WhatsApp). How do you do this with Supabase queries when the last message timestamp is stored in the conversations table? Try it before asking.
Phase 4 · Week 2 · Days 4–7

Messages

Goal: Send, receive, edit, soft-delete messages. Pagination. Mark as seen.

Two critical decisions: cursor-based pagination and soft delete. Both affect your app's quality significantly.

Cursor vs OFFSET Pagination

OFFSET (Wrong at Scale) OFFSET 10000 LIMIT 20 — database scans 10,020 rows to return 20. Slows down dramatically as conversations grow.
CURSOR (Always Fast) WHERE id < :lastId ORDER BY id DESC LIMIT 20 — jumps directly via index. Same speed at 100 or 10 million rows.

sendMessage — More Than Just INSERT

  1. Validate: content not empty, user is a member of this conversation
  2. Insert message into messages table
  3. Update conversation's last_message_preview and last_message_at
  4. Increment unread_count for ALL members EXCEPT the sender
  5. Return the created message
  • 🔴 CriticalsendMessage — validate, insert, update conversation metadata, increment unread
  • 🔴 CriticalgetMessages — cursor-based pagination, default limit 30, filter out soft-deleted
  • 🔴 CriticaldeleteMessage — soft delete only (set deleted_at = now()), own messages only
  • 🟡 ImportantmarkAsSeen — mark seen + reset unread_count to 0 for current user
  • 🟡 ImportanteditMessage — edit own message only, within 15 min of creation, set is_edited: true
  • 🟢 NiceTest in Postman then git commit: feat: add message endpoints with pagination and soft delete
Challenge — Phase 4 In markAsSeen, you reset unread_count to 0. But what if two users open the same conversation simultaneously and both call this endpoint at the same time? Is there a race condition? Think about it — this is a real-world concurrency problem.
Fields to add to messages table deleted_at TIMESTAMP · is_edited BOOLEAN · edited_at TIMESTAMP · file_url TEXT · file_type TEXT

Groups

Goal: Full group management with admin controls

Groups are conversations with extra metadata and an admin. The key new concept: always verify admin before any destructive action.

// Admin check — do this FIRST in every admin-only function
const { data: group } = await supabase
  .from('groups').select('admin_id').eq('id', groupId).single()
if (group.admin_id !== req.user.id)
  return sendResponse(res, 403, 'Only the group admin can do this')
Edge Cases to Handle What if admin tries to remove themselves? · What if last member leaves (auto-delete group?)· What if someone tries to add a user already in the group? · What if a non-member requests group details? Handle all four.
  • 🔴 CriticalcreateGroup — create group, add creator as admin, add initial members
  • 🔴 CriticalgetGroupDetails — group info + member list, members only (403 for non-members)
  • 🟡 ImportantupdateGroup — update name/avatar, admin only (admin check first)
  • 🟡 ImportantaddMember — admin only, check for duplicate before inserting
  • 🟡 ImportantremoveMember — admin only, cannot remove self via this endpoint
  • 🟡 ImportantleaveGroup — non-admin only (admin must transfer role first)

Common Mistakes — Supabase Edition

1. Multiple Supabase client instances

Never call createClient() inside a controller. Create once in config/supabase.js, import everywhere.

2. Exposing the service_role key

service_role bypasses ALL RLS. Never send it to frontend. Never commit it. Server-side admin operations only.

3. Broadcasting socket messages before saving to DB

Always: save to Supabase FIRST → then broadcast. Broadcast first = ghost messages on refresh.

4. Using OFFSET pagination for messages

OFFSET 50000 LIMIT 20 scans 50,020 rows. Use cursor-based pagination from Phase 4.

5. Permanently deleting messages

Breaks unread counts, conversation previews, and history for other users. Always soft delete.

6. Skipping admin check before destructive actions

Always check req.user.id === group.admin_id before delete, remove member, update group.

7. Not updating conversation metadata after sending message

After every message: update last_message_preview, last_message_at, increment unread_count.

8. Forgetting the offline reconnect scenario

When user comes back online, they missed socket events. Frontend must call getMessages REST API. Your REST API is the safety net for socket failures.
